# PRODUCT_RULES.md

# AI 产品交付工作台产品规则 V2

> 本文档用于约束 AI 产品交付工作台的生成逻辑、展示逻辑和交付物标准。
>
> 它不是功能需求文档，而是产品规则文档。
>
> 后续无论是 MockAIProvider、真实大模型 Prompt、线框图渲染器、PRD 生成、研发说明生成，都必须遵守这些规则。

---

## 0. 产品定位总规则

本产品不是：

- AI UI 生成器
- 低代码平台
- 应用生成器
- 代码生成器
- 高保真设计稿生成器
- Figma / Axure / 墨刀替代品

本产品是：

> AI 产品交付工作台。

核心目标是：

> 把混乱页面、截图、想法、AI 生成页面、竞品页面，整理成可评审、可交付、研发能接手的产品资产。

所有功能、页面、Prompt、线框图、PRD、研发说明，都必须围绕三个问题判断：

1. 是否帮助用户更快评审？
2. 是否帮助用户更好交付？
3. 是否帮助研发更快接手？

如果某个功能不能服务这三个目标，MVP 阶段不做。

# 1. 真实页面与页面状态规则

## 1.1 ProductPage 只表示真实页面

`ProductPage` 只表示真实存在的页面，例如：

- 首页
- 医生列表页
- 医生详情页
- 预约确认页
- 我的预约页
- 个人中心

不要把以下内容加入 `pageList`：

- 弹窗
- 底部弹层
- Toast
- 表单错误状态
- 加载状态
- 空状态
- 筛选结果状态
- Tab 切换状态
- 授权弹窗
- 支付确认状态

这些不是独立页面，而是页面内状态。

---

## 1.2 WireframePage 表示真实页面的状态帧

`WireframePage` 不是只能一页对应一张图。

正确规则是：

> 一个真实页面可以对应多个 WireframePage 状态帧。

例如：

```txt
医生列表页
├── 默认状态
├── 科室筛选底部弹层打开
├── 筛选结果状态
└── 无医生结果空状态
```

这些状态帧都属于同一个真实页面。

---

## 1.3 同一真实页面的状态帧 pageId 必须相同

例如：

```ts
{
  id: "doctor-list-default",
  pageId: "doctor-list",
  pageName: "医生列表页 - 默认状态",
  blocks: []
}

{
  id: "doctor-list-filter-sheet",
  pageId: "doctor-list",
  pageName: "医生列表页 - 科室筛选底部弹层打开",
  blocks: []
}
```

规则：

- `id` 必须唯一
- `pageId` 可以相同
- 相同 `pageId` 表示这些状态帧属于同一个真实页面
- `pageName` 必须体现当前状态

# 2. 页面状态帧生成规则

## 2.1 必须拆分状态帧的情况

只要是：

> 不跳转新页面，但在原页面上发生了明显视觉变化

就必须生成独立状态帧。

包括：

- 弹窗 modal
- 底部弹层 bottom-sheet
- 抽屉 drawer
- Toast / Snackbar
- 表单校验错误
- 加载中状态
- 提交中状态
- 操作成功状态
- 操作失败状态
- 空状态
- 异常状态
- 筛选结果状态
- 搜索结果状态
- Tab 切换后内容变化
- Picker / Dropdown / Popover
- 授权弹窗
- 支付确认
- 订阅消息提示

---

## 2.2 不应该拆分状态帧的情况

以下不生成状态帧：

- hover
- focus
- active
- 按钮按下瞬间
- 普通动画过程
- 纯装饰变化
- 页面跳转行为

---

## 2.3 页面跳转不属于状态帧

如果用户操作会跳转到新页面，例如：

```txt
点击医生卡片 → 医生详情页
```

这不应该生成：

```txt
首页 - 医生卡片点击状态
```

页面跳转应该由 `flows` 表达，不应该作为 `wireframes` 的状态帧。

正确方式：

```txt
flows:
首页 / 点击医生卡片 / 医生详情页
```

错误方式：

```txt
wireframes:
首页 - 医生卡片点击（跳转详情）
```

# 3. Wireframe 树结构规则

线框图必须表达树结构，而不是平铺页面列表。

正确层级：

```txt
真实页面
├── 状态帧
│   ├── 模块
│   │   ├── 字段
│   │   └── 操作
```

例如：

```txt
医生列表页
├── 默认状态
│   ├── 顶部导航
│   ├── 搜索框
│   ├── 科室筛选入口
│   └── 医生卡片列表
│       └── 医生卡片
│           ├── 医生头像
│           ├── 医生姓名
│           ├── 职称
│           ├── 擅长领域
│           └── 预约按钮
│
├── 科室筛选底部弹层打开
│   ├── 原页面主要模块
│   └── 科室筛选底部弹层
│       ├── 科室选项
│       ├── 职称筛选
│       └── 确认筛选按钮
```

# 4. 弹窗 / 底部弹层 / Toast 规则

## 4.1 弹窗不是页面

弹窗不进入 `pageList`。

弹窗应该作为当前页面的状态帧出现。

例如：

```txt
我的预约页 - 默认状态
我的预约页 - 取消预约确认弹窗
```

---

## 4.2 底部弹层不是页面

底部弹层也不进入 `pageList`。

例如：

```txt
医生列表页 - 默认状态
医生列表页 - 科室筛选底部弹层打开
```

---

## 4.3 Toast 不是弹窗

Toast 是轻提示，不应该渲染成大弹窗。

如果当前 `WireframeBlockType` 没有 `toast`，可以用：

```ts
type: "text"
```

或：

```ts
type: "card"
```

来表达 Toast 状态。

不要用：

```ts
type: "modal"
```

表达 Toast。

---

# 5. 平台外框与业务导航规则

## 5.1 平台只决定外框，不决定业务导航

平台只决定设备环境，不决定业务结构。

Web：

- 可以有桌面浏览器外框
- 可以有桌面工作台容器
- 但不代表一定有侧边栏

H5：

- 可以有手机网页外框
- 可以有移动网页标题栏
- 但不代表一定有底部固定按钮或底部导航

App：

- 可以有状态栏
- 可以有 App 导航栏
- 但不代表一定有底部 Tab

小程序：

- 可以有小程序导航栏
- 可以有右上角胶囊按钮
- 但不代表一定有底部 TabBar

也就是说：

- 小程序可以没有底部 TabBar
- App 可以没有底部 Tab
- Web 可以没有侧边栏
- H5 可以没有底部固定按钮

不要因为 `platform` 是 `mini-program` / `app` / `web` / `h5`，就硬加业务导航。

---

## 5.2 业务导航必须来自数据

只有 `wireframe.blocks` 里真实存在对应的 `nav` / `tabs` block，才渲染业务导航。

例如只有存在：

```ts
{
  type: "tabs",
  title: "底部 TabBar",
  actions: ["首页", "找医生", "预约", "我的"]
}
```

才渲染底部 TabBar。

如果没有这个 block，就不渲染底部 TabBar。

不要硬编码：

```txt
首页 / 找医生 / 预约 / 我的
首页 / 医生 / 预约 / 我的
首页 / 医生管理 / 预约管理 / 用户管理 / 系统设置
```

这些只能来自数据，不能由设备层或渲染器凭空创建。

---

## 5.3 TabBar 必须来自 bottomTabBlock

底部 TabBar 必须来自 `wireframe.blocks` 中明确存在的 bottomTabBlock。

判断可以基于：

- `block.type === "tabs"`
- title 包含：底部、TabBar、底部导航、底部 Tab
- 或 actions 明显是主导航，例如：首页、找医生、预约、我的

如果找到 bottomTabBlock：

- 设备层使用 `bottomTabBlock.actions` 渲染 TabBar
- 内容区过滤掉这个 bottomTabBlock，避免重复

如果没有找到 bottomTabBlock：

- 不渲染 TabBar
- 不要生成默认 TabBar
- 不要补齐导航项

---

## 5.4 Web 侧边栏必须来自 sidebar nav block

Web 侧边栏必须来自明确的 `nav` block。

判断可以基于：

- `block.type === "nav"`
- title 明确包含：侧边栏、菜单、功能导航、页面导航、管理菜单、左侧导航

如果找到 sidebarBlock：

- 设备层渲染侧边栏
- 内容区过滤掉这个 sidebarBlock，避免重复

如果没有找到 sidebarBlock：

- 不渲染侧边栏
- 不要因为 platform 是 web 就强加侧边栏

不要仅凭 `actions.length >= 3` 判断为侧边栏。  
因为服务入口、分类导航、状态筛选也可能有 3 个以上选项。

---

## 5.5 普通导航必须保留在内容区

以下内容属于页面内容，不要被误过滤：

- 服务入口
- 快捷入口
- 功能入口
- 应用入口
- 科室 Tab
- 状态 Tab
- 筛选 Tab
- 分类 Tab
- 日期 Tab

这些不是设备导航，不应该被设备层抢走。

---

## 5.6 内容区只过滤设备层已经渲染过的 block

内容区不要简单过滤所有 `nav` 或所有 `tabs`。

推荐规则：

```ts
const bottomTabBlock = getBottomTabBlock(wireframe.blocks);
const sidebarBlock = getWebSidebarBlock(wireframe.blocks);

const contentBlocks = wireframe.blocks.filter(block =>
  block.id !== bottomTabBlock?.id &&
  block.id !== sidebarBlock?.id
);
```

只过滤设备层已经渲染过的 block。  
不要过滤服务入口、科室 Tab、状态 Tab、筛选 Tab。

# 6. 数据数量规则

## 6.1 数据返回几个，就展示几个

所有展示数量必须由数据决定。

例如：

- TabBar actions 返回 1 个，就展示 1 个
- TabBar actions 返回 2 个，就展示 2 个
- TabBar actions 返回 4 个，就展示 4 个
- 侧边栏 actions 返回几个，就展示几个菜单
- 服务入口 actions 返回几个，就展示几个入口
- 表单 fields 返回几个，就展示几个表单项
- 弹窗 actions 返回几个，就展示几个按钮
- 底部弹层 actions 返回几个，就展示几个按钮
- 列表 children 返回几个，就展示几个子项

不要为了好看自动补齐。

---

## 6.2 示例数据只能用于字段值展示，不能用于补结构

允许：

```txt
字段：医生姓名
展示：张医生
```

允许：

```txt
字段：手机号
展示：138****8888
```

允许：

```txt
字段：商品价格
展示：¥99.00
```

不允许：

```txt
没有底部菜单数据，却自动生成 4 个底部菜单
没有侧边栏数据，却自动生成侧边栏
没有 3 个医生卡片，却自动生成 3 个医生卡片
没有服务入口数据，却自动生成 4 个服务入口
```

---

## 6.3 actions 为空时不补按钮

如果 `actions` 为空：

- 卡片不显示按钮
- 表单不显示按钮
- 弹窗不显示按钮区
- 底部弹层不显示按钮区
- 不要自动补“确认 / 取消 / 预约 / 改约 / 详情”

---

## 6.4 fields 为空时不补字段

如果 `fields` 为空：

- 表单不显示表单项
- 信息卡片不补字段
- 表格不补字段列
- 不要自动补身份证号、症状描述、预约时间、手机号等字段

---

## 6.5 children 为空时不补多个业务卡片

如果 `children` 为空：

- 不要自动补 2 个或 3 个真实业务卡片
- 可以显示模块标题和说明
- 最多显示 1 个弱占位示例
- 不要把弱占位伪装成真实业务数据

---

# 8. 线框图展示目标规则

线框图不是字段清单，也不是最终高保真 UI。

线框图目标是：

> 带示例数据的中保真静态页面草图。

它应该做到：

- 像用户真的打开了这个页面
- 能看到页面结构
- 能看到模块位置
- 能看到字段承载的数据
- 能看到按钮位置
- 能看到弹窗 / 底部弹层 / Toast 状态
- 但不追求最终视觉设计

---

## 8.1 不要只展示字段名

错误：

```txt
医生卡片
[头像] [姓名] [职称] [擅长]
[预约]
```

正确：

```txt
圆形头像占位
张医生
主任医师 · 中医内科
擅长：脾胃调理、针灸、失眠
评分 4.9 · 已预约 326 次
[预约]
```

---

## 8.2 字段应该展示示例数据

字段不是用来直接展示字段名的。

字段应该用于推断页面上展示的数据。

---

# 9. 中保真静态原型规则

## 9.1 医生卡片

只有医疗语义下，且 block 表示医生卡片、单个医生、医生资料卡片、专家卡片、医师卡片时，才渲染为医生卡片。

不要把这些误判为单个医生卡片：

- 推荐医生
- 医生列表
- 医生信息
- 医生管理
- 医生筛选

---

## 9.2 预约记录卡片

如果 block 表示预约记录、我的预约、预约卡片，应渲染为预约记录卡片。

结构：

- 医生姓名或服务名称
- 科室 / 职称或服务类型
- 预约时间
- 预约人
- 状态
- 操作按钮

按钮必须来自 actions。  
如果 actions 为空，不要自动补“取消 / 改约 / 详情”。

---

## 9.3 表单

表单应渲染为真实表单，而不是字段堆叠。

fields 有几个，就展示几个。  
没有 fields，不补字段。

如果是表单校验错误状态：

- 错误字段边框应有错误提示
- 下方显示错误文案，例如“请填写必填信息”

---

## 9.4 列表

列表不应该只显示：

```txt
列表项 1
列表项 2
列表项 3
```

列表应根据语义渲染：

- 医生列表：渲染医生卡片
- 商品列表：渲染商品卡片
- 课程列表：渲染课程卡片
- 简历列表：渲染简历卡片
- 订单列表：渲染订单卡片
- 内容列表：渲染内容卡片
- 通用列表：渲染普通信息列表

列表数量由 children 决定。  
没有 children 时，不默认补 3 个真实业务卡片。

---

## 9.5 Banner

Banner 不应只是一个写着“Banner”的色块，也不能默认写死医疗文案。

Banner 应根据行业 / 页面 / 模块语义展示示例内容。

无法判断行业时使用通用 Banner：

```txt
核心功能入口
这里展示该页面的核心信息和操作入口
```

---

## 9.6 服务入口

服务入口必须来自 nav block 的 actions。

actions 有几个，就显示几个入口。

不要自动补：

```txt
找医生 / 预约挂号 / 健康档案 / 我的预约
```

除非这些来自 actions。

---

## 9.7 TabBar

TabBar 必须来自 bottomTabBlock.actions。

actions 有几个，就显示几个 Tab。

不要自动补齐。

---

## 9.8 弹窗

弹窗应像真实弹窗。

弹窗按钮来自 actions。

如果 actions 为空，不显示按钮区。  
不要默认补“取消 / 确认”。

---

## 9.9 底部弹层

底部弹层应像真实底部弹层。

fields 用于选项列表。  
actions 用于底部操作按钮。

不要把 actions 同时渲染成选项和按钮。

如果 fields 为空，不显示选项列表。  
如果 actions 为空，不显示按钮区。

---

## 9.10 Toast

Toast 应像轻提示，例如：

```txt
预约成功，请按时就诊
```

Toast 不应渲染成大弹窗。  
Toast 不应拦截真实页面点击。

---

# 10. 平台差异规则

## 10.1 Web

Web 线框图不使用手机壳。

Web 应表现为桌面端页面：

- 浏览器 / 工作台容器
- 顶部导航
- 内容区
- 筛选区
- 表格
- 弹窗表单

如果没有 sidebarBlock，不显示侧边栏。

---

## 10.2 H5

H5 应表现为手机网页：

- 手机网页容器
- 顶部标题栏
- 纵向滚动内容
- 卡片列表
- 表单
- 底部固定按钮

如果没有 bottomTabBlock，不显示底部导航。

---

## 10.3 App

App 应表现为原生移动端：

- 状态栏
- 原生导航栏

如果没有 bottomTabBlock，不显示底部 Tab。

Action Sheet 使用 bottom-sheet 表达。  
原生弹窗使用 modal 表达。

---

## 10.4 小程序

小程序应表现为小程序页面：

- 小程序导航栏
- 右上角胶囊按钮

如果没有 bottomTabBlock，不显示 TabBar。

---

# 11. WireframeBlock 类型规则

当前允许的 `WireframeBlock.type`：

```txt
header
nav
banner
card
list
form
button
tabs
modal
bottom-sheet
table
empty-state
text
```

不要新增非法类型：

```txt
sidebar
action-sheet
content
pagination
toast
```

替代规则：

| 想表达 | 使用类型 |
|---|---|
| sidebar | nav |
| action-sheet | bottom-sheet |
| content | card |
| pagination | nav 或 text |
| toast | text 或 card |

---

# 12. PRD 同步规则

如果线框图生成了页面状态帧，PRD 必须体现这些状态。

PRD 中至少应该说明：

- 页面默认状态
- 页面内弹窗状态
- 页面内底部弹层状态
- Toast 状态
- 表单校验错误状态
- 加载状态
- 空状态
- 异常状态

PRD 不应该只写真实页面，不写页面内状态。

---

# 13. 研发说明同步规则

如果线框图中存在页面状态帧，研发说明也必须体现。

研发说明应说明：

- 哪些是真实页面
- 哪些是页面内状态
- 弹窗如何触发
- 底部弹层如何触发
- Toast 何时出现
- 表单错误如何校验
- 加载 / 空 / 异常状态如何处理
- 哪些接口会影响状态切换

研发说明不能只是 PRD 的重复。

---

# 14. 生成与展示职责边界

## 14.1 生成层负责

生成层负责输出：

- 真实页面
- 页面状态帧
- 模块
- 字段
- 操作
- 弹窗 / 底部弹层 / Toast 状态
- PRD
- 研发说明

---

## 14.2 展示层负责

展示层负责把生成结果渲染成：

- 页面状态树
- 中保真静态原型
- 设备外框
- 示例数据
- 真实页面布局感

展示层可以根据字段名生成示例数据，但不应改变原始数据结构。

展示层不能用示例数据补结构数量。

---

# 15. 静态原型点击规则

线框图设备内部的按钮都是静态展示，不需要点击。

真实页面按钮才需要点击，例如：

- 返回跳转关系
- 保存线框图
- 下一步生成 PRD
- 左侧页面状态树切换

modal / bottom-sheet / toast 等线框图内部元素不应拦截真实页面点击。

---

# 16. 不要做的事情

MVP 阶段不要做：

- 高保真 UI 设计
- 真实代码生成
- 真实 App / 小程序 / H5 工程生成
- Figma 导出
- Axure 导出
- 墨刀导出
- 拖拽式编辑器
- 复杂缩放画布
- 复杂组件库编辑
- 图片级视觉还原
- 渲染器凭空补业务结构
- 所有行业默认套用医疗示例数据

这些会让产品偏离“产品交付工作台”的定位。

---

# 17. 关键验收标准

线框图相关能力必须满足：

1. 真实页面与页面状态分清楚。
2. 状态帧不进入 pageList。
3. 页面跳转不生成状态帧。
4. 弹窗 / 底部弹层 / Toast / 表单错误 / 加载 / 空状态能独立展示。
5. 左侧导航是“真实页面 → 状态帧”的树结构。
6. 中间预览像带示例数据的中保真静态原型。
7. 字段不只是字段名，而是展示示例数据。
8. 平台只决定外框，不决定业务导航。
9. 业务导航必须来自数据。
10. 数据返回几个，就展示几个。
11. 示例数据只用于字段值展示，不能补结构。
12. 示例数据必须行业感知，不能默认医疗。
13. PRD 能说明页面内状态。
14. 研发说明能说明状态如何开发。
15. 不生成真实代码。
16. 不偏离产品交付工作台定位。

---

# 18. 一句话原则

> 页面结构要清楚，页面状态要拆开，字段要展示示例数据，平台只决定外框，业务结构和数量由数据决定，示例数据必须行业感知，线框图要像用户真的打开页面时看到的中保真静态草图。
